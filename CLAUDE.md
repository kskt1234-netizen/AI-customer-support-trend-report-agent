# CLAUDE.md — 프로젝트 헌법 (System Constitution)

이 문서는 AI 어시스턴트(그리고 사람 기여자)가 이 저장소에서 작업할 때 지켜야 할
**핵심 철학, 불변 원칙, 검증 가드레일**을 정의한다. 절차 대본이 아니라 판단 기준이다.
세부 구현 방법은 자율적으로 결정하되, 아래 원칙과 충돌하면 원칙이 이긴다.

## 프로젝트 정체성

B2B SaaS 고객지원을 가정한 **마스터-워커 에이전트 허브** (LangGraph 기반).
마스터(`src/main_graph.py`)는 의도를 분류해 라우팅만 하고, 실제 업무는
워커 서브그래프(`src/sub_graphs/`)가 수행한다. 목표는 "돌아가는 데모"가 아니라
**프로덕션 수준의 설계·방어·검증을 코드로 증명하는 포트폴리오**다.

## 불변 원칙 (Architecture Invariants)

이 원칙을 깨는 변경은 "리팩토링"이 아니라 "아키텍처 변경"이며, 인간 승인 없이 금지된다.

1. **마스터-워커 분리 (SRP)**
   - 마스터 그래프의 책임은 오직 라우팅. 업무 로직을 마스터에 넣지 않는다.
   - 새 기능은 새 워커 서브그래프로 추가한다. 마스터에는 노드 등록 + 라우팅 분기
     한 줄씩만 늘어나야 정상이다.

2. **LLM 공급사 추상화 (DIP)**
   - 노드는 `LLMProvider`(ABC, `src/llm/base.py`) 계약에만 의존한다.
   - `ChatOpenAI` 등 구체 클래스의 import는 해당 공급사 구현 파일
     (`src/llm/openai_provider.py`) 안에만 존재할 수 있다. 다른 파일에서 발견되면 위반.
   - 공급사 선택은 `src/llm/factory.py`의 레지스트리 한 곳에서만 결정한다(OCP).
   - 새 공급사 추가 = 구현 파일 1개 + 레지스트리 1줄. 노드/그래프 코드는 0줄 수정.

3. **LLM은 언어, 코드는 연산**
   - 수치 계산(성장률 등)·형식 검사·라우팅 분기는 결정론적 파이썬 코드로 한다.
     LLM에게 산수를 시키는 코드는 어떤 이유로도 금지.
   - LLM은 자유 텍스트 생성(초안·답변)과 의미 판단(맥락·톤 검수)에만 쓴다.

4. **경계는 Pydantic, 내부는 TypedDict**
   - 그래프 경계를 넘는 데이터(LLM 구조화 출력, 최종 산출물)는 `src/schemas.py`의
     Pydantic 모델로 런타임 검증을 강제한다.
   - 노드 사이를 흐르는 내부 상태는 `src/state.py`의 TypedDict로 가볍게 유지한다.
   - 새 필드는 반드시 `state.py`에 타입과 함께 선언한다(암묵적 필드 금지).

5. **모든 루프에는 탈출 장치**
   - 그래프에 사이클(재시도 루프 등)을 만들 때는 반드시 카운터 상한과
     탈출 경로(`bail_out` 류)를 함께 만든다. 상한 없는 루프는 API 요금 폭탄이다.

## 코드 작성 규칙

- **LLM 호출은 실패한다고 가정하라.** 네트워크 장애·레이트리밋·타임아웃은 예외 상황이
  아니라 일상이다. LLM을 호출하는 모든 노드는 예외를 잡아 **우아하게 저하(graceful
  degradation)** 해야 한다: 크래시 대신 안전한 기본 경로 또는 `error` 필드 기록.
  단, 예외를 조용히 삼키지 말 것 — 반드시 `logger.exception`/`warning`으로 남긴다.
- **print 금지, logging 사용.** 로거는 `src/logging_config.py`의 `get_logger(__name__)`로
  얻는다. 운영 판단에 필요한 결과는 INFO, 디버깅 근거(reasoning 등)는 DEBUG.
  (`scripts/`의 수동 실행 스크립트는 사용자 대면 출력이므로 print 허용.)
- **노드는 '바꿀 필드만' 담은 dict를 반환한다.** State 전체를 재구성해 반환하지 않는다.
- **라우터 함수는 순수 함수로 유지한다.** State를 읽기만 하고, 부작용 없이 분기 키
  문자열만 반환해야 단위 테스트가 가능하다.
- **프롬프트는 모듈 상수로 분리한다.** 로직과 섞지 않아야 골든셋 평가를 보며
  프롬프트만 반복 튜닝할 수 있다.

## 검증 가드레일 (Self-Verification — 위반 시 작업 미완료로 간주)

1. **코드를 추가/수정한 후에는 사용자에게 보고하기 전에 반드시 스스로 테스트를 실행한다.**
   ```
   .venv\Scripts\python.exe -m pytest -v        # Windows (이 저장소 표준)
   ```
   실패가 있으면 원인을 찾아 수정하고 재실행한다. **100% 통과를 확인하기 전까지
   "완료"라고 말하지 않는다.**
2. **새 동작에는 새 테스트.** 방어 로직·분기·연산을 추가했으면 그 동작을 검증하는
   테스트를 `tests/`에 함께 추가한다. 테스트 없는 방어 코드는 없는 코드다.
3. **투트랙 테스트 구조를 훼손하지 않는다.**
   - 자동 테스트(`tests/`)는 mock 모드에서 **API 키 없이, 비용 0으로, 결정론적으로**
     통과해야 한다. 실제 API를 호출하는 테스트를 자동 수집 경로에 넣지 않는다.
   - 실제 API 검증은 `TEST_MODE=real`(수동) 또는 `scripts/run_e2e.py`(수동)의 몫이다.
4. **골든셋(`tests/test_cases.json`)은 분류기의 스펙이다.** 분류 프롬프트를 바꿨으면
   골든셋 통과를 확인하고, 새 경계 케이스를 발견했으면 골든셋에 추가한다.

## HITL (인간 승인) 규칙

- 다음은 **반드시 인간의 확인·승인 후에만** 진행한다:
  - 위 불변 원칙을 깨거나 코어 아키텍처(그래프 위상, 추상화 계층)를 크게 바꾸는 변경
  - 최종 릴리즈/마무리 단계 (커밋·푸시 포함 — 명시적 요청 없이 커밋하지 않는다)
  - 유료 API를 호출하는 작업의 자동화(비용 발생 지점의 신설)
- 그 외 세부 구현·리팩토링·테스트 추가·문서화는 자율적으로 진행하되,
  결과와 근거(테스트 출력)를 함께 보고한다.

## 비밀 관리

- `.env`는 절대 커밋하지 않는다. 키는 로컬 `.env`와 GitHub Secrets에만 존재한다.
- 코드·문서·로그 어디에도 실제 API 키를 쓰지 않는다. 예시는 `.env.example`처럼
  플레이스홀더로만.
